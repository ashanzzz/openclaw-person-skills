# 吉众对账匹配模板（A/B 双口径）

当用户要做“财务明细 ↔ ERPNext 采购发票/账本对账”、并要求 **同时保留**：
- A：银行/现金**实际出账**（含手续费、可能是部分付款）
- B：**凭证/发票口径**（用于与发票金额勾稽：扣手续费 + 可包含挂应付）

使用本技能。

## Inputs
- `YYYYMM`（期间，如 `202601`）
- DSM（Synology）WebDAV（从 `secure/api-fillin.env` 读取 `SYNOLOGY_HOST_LAN` + `SYNOLOGY_WEBDAV_(HTTPS|HTTP)_PORT` + `SYNOLOGY_WEBDAV_USER/PASS`；Nextcloud 已弃用）
- 上传路径：脚本会自动生成默认目录（`homes/<webdav_user>/财务/对账/<公司>/<YYYYMM>/`），也可用 `--remote-dir` 覆盖
- ERPNext API（从 `secure/api-fillin.env` 读取 `ERP_BASE_URL/ERP_API_TOKEN`）
- 财务明细 xls（例如 `1月明细_202601.xls`）

## Output
- 生成并上传：`匹配模板_<YYYYMM>_自动填充_v16_AB_含规格备注_含入账_含余额_含应付_含财务应付_拆分无票_发票含电汇手续费_财务表删凭证号_不含结转行.xlsx`
  - Sheet `财务银行` / `财务现金`（支出）：
    - A/B 双口径、手续费、B 的剩余匹配金额（`挂应付`列已删除，避免误解）
    - 新增列：`摘要`
    - 结转余额核对：见 sheet `余额汇总`（上期结转余额 / 期末结转余额）
  - Sheet `财务银行入账` / `财务现金入账`：银行/现金 **入账（收入）** 明细（用于对账补全）
  - Sheet `财务应付`（支出/仅应付）：与 `财务银行/财务现金` **同样的匹配逻辑**，但来源是“仅应付(A=0)”的凭证（从 `应付账款` 贷方提取），用于把“无出账但已挂应付”的发票也纳入匹配体系。
  - Sheet `发票基准`（仅有发票号）：来自 ERPNext 采购发票+明细行，过滤规则：`bill_no` 不是 `0` / `00` / `00/00000` 这类全 0 形式；并写入：
    - `单据日期`（`Purchase Invoice.posting_date`）
    - `发票号`（`Purchase Invoice.bill_no`）
    - `电汇手续费`：取匹配到的财务凭证的手续费（560302/财务费用-手续费用）
    - 规格：`items.custom_guige_xinghao`
    - 备注：优先 `items.custom_备注`，否则用 `invoice.custom_明细摘要`/`invoice.remarks`
  - Sheet `无发票支出`：来自 ERPNext 采购发票+明细行，过滤规则：`bill_no` 为空或全 0（如 `0`、`00/00000` 等）
  - Sheet `财务银行` / `财务现金` / `财务应付`：
    - 列已删除：`财务凭证号`
    - 不再写入 `上期结转/本期结转` 行（避免与“本表仅支出、收入不在表内”产生理解混乱）
  - Sheet `余额汇总`：银行/现金 上期结转余额 + 期末结转余额（用于快速核对）
- 同目录额外保存（给 Zero 做后续匹配/你自己也可查）：
  - `bank_cash_income_<YYYYMM>.csv`（银行+现金入账合并表）

## Notes / Guardrails
- 本技能只做“对账工作表生成 + 回传”，**不创建/提交** ERPNext 付款单、不改主数据。
- 兼容轻微格式差异：如 sheet 名从 `应付账款` 变为 `03应付账款`（带前缀数字），脚本会智能归一化后再做匹配。
- “财务凭证号”与“财务记账名”重复时：保持凭证号列可为空（不破坏用户表格习惯）。
- A/B 口径定义：
  - A = 银行/现金 sheet 的贷方合计（真实出账）
  - B = 用于勾稽发票的可匹配金额：
    - 若存在非银行/现金借方（除手续费外）或存在挂应付：B 会扣除手续费并可合并挂应付
    - 若仅手续费分录（如短信费）：不把手续费从 B 中扣掉，避免 B 变 0

## Matching Strategy (固化思考方法, 2026-03-23)

> 核心目标：宁可少匹配/留空，也不要“匹配错”。

1) **旧表的“财务凭证”只能当线索**（低置信）
- 不能仅因为旧列写了 `银行-0008` 就认定它对。
- 旧线索只能用于“减少搜索空间 / 多解时排序”，**不得覆盖金额的唯一解**。

2) **金额优先（硬规则，用户要求：金额最重要）**
- 金额约束是硬门槛：
  - 银行：`cap = 银行出账(A) - 手续费`，要求匹配后满足：`Σ电汇金额(不含手续费) ≤ cap`，且 `财务银行 剩余匹配金额 = cap - 已匹配金额`。
  - 现金：`cap = 记账金额(B)`
  - 应付：`cap = 记账金额(B)`
- 组合匹配（subset-sum / n-sum）以“凑整闭合”为第一目标：只要金额存在 **唯一解**，可在不依赖摘要的情况下自动匹配。

2.1) **摘要/关键词（软约束，仅用于消歧与解释）**
- 摘要不作为硬门槛；只用于：
  - 多解时的排序（选择更符合语义的组合）
  - 候选爆炸时的软过滤（加速搜索），但不能因此漏掉金额唯一解
- 银行手续费格点（仅限财务银行的灵活校验/兜底）：
  - 若手续费列缺失/异常，可允许“剩余差额”落在常见手续费集合（如 4.5 的倍数，或当月出现过的手续费值集合），作为凑整的额外约束。

3) **分配顺序（避免把应付/电汇写反）**
- 先匹配 **电汇/现金（支付）** 的高置信项。
- 再对“剩余未匹配金额”去匹配 **应付** 的高置信项。
- 不允许用“差额=应付”硬推；只有找到明确的 `应付-xxxx` 凭证才写入。

4) **组合凑整 / 残差补齐（Composite voucher residual fill）**

4.1 银行/现金：**单票残差补齐**
- 典型场景：同一笔银行凭证（如 `银行-0001`）一次性支付了 **多张发票**（例如电费+水费），财务摘要可能只写了其中一种。
- 规则：当某个银行凭证已经高置信匹配了一张发票后，若该凭证还有 `capRemain = (A出账-手续费)-已匹配金额` 的**剩余**，应尝试用“剩余金额=某张未匹配发票金额”的方式补齐。
- 为避免误配，只在满足以下条件时自动补齐：
  - 该凭证 **已经** 匹配过至少 1 张发票（说明它可能是合并支付）。
  - 只找到 **唯一** 候选发票（金额精确相等 ±0.01）。
  - 候选发票与该凭证（或该凭证已匹配的发票）属于同一宽松类别（例如“公用事业”簇：电费/水费/燃气/物业等），或可解释为同类合并支付。
  - 若存在多个候选，宁可留空等待人工确认。

4.2 应付：**多票凑整（n-sum / 子集和匹配）**
- 典型场景：`应付-0027` 金额 421.82 已匹配一张发票 327.38 后，剩余 94.44 = 88.44 + 6.00（两张发票合计）。更一般地，可能是 **n 张发票**合计凑成同一条应付凭证的剩余。
- 这类匹配方式叫：**凑整匹配 / 组合匹配 / 子集和匹配（subset-sum；2-sum 是 n-sum 的特例）**。

规则（固化）：当某条应付凭证已匹配过至少 1 张发票且还有剩余时：
1) 先做 **2-sum**（最快、最稳）：在未匹配集合中找“唯一一对”金额和 = 剩余（±0.01）。
2) 若 2-sum 无解，再做 **n-sum（有界子集和）**：
   - 候选集必须先“强过滤”缩小（否则组合爆炸）：
     - 类别簇过滤：优先限定在同类/同摘要簇（办公耗材、快递、低值易耗等）。
     - 关键词过滤：摘要核心词/关键词与发票侧文本（物料名+规格+备注）至少弱命中。
       - 允许小范围“同义词/词根”归一（例如 `修理`≈`维修`，`物流/收派服务`≈`快递`，`无尘布`≈`无纺布`），避免明显同类却漏配。
     - 金额过滤：候选金额不得大于剩余；过小到纯噪声的条目可先排除。

   - **分层候选（重要）**：按置信度分 3 层尝试子集和，保证“先严后松”：
     1) strict：仅允许关键词命中（最可靠）
     2) cluster：同类别簇允许（如办公耗材簇允许“无尘布/复印纸/硒鼓”等混合；耗材簇允许小额快递费作为附件）
     3) relaxed：当且仅当 strict/cluster 无解时，允许更宽松的候选，但仍必须满足 **唯一解**；对“收派服务/快递”类凭证，要求组合内至少包含 1 条快递/物流类发票作为解释锚点。
   - 搜索算法：对候选金额排序（通常从大到小），用 DFS/回溯 + 剪枝（当前和>剩余即剪、即使全加也达不到则剪）。
   - **有界**：默认只尝试子集大小 ≤ 5（或用户指定上限），并设置超时/步数上限，避免卡死。
   - **唯一性门槛**：
     - 只有在找到“唯一解（唯一一组发票组合）”时才自动写入；
     - 若找到多解/歧义解 → 停止自动匹配并留空，交给人工确认。
3) 一旦找到唯一解：把该组合内每张发票写入同一个 `应付-xxxx`，并把应付行的 `已匹配金额/剩余匹配金额` 更新。

4) **手续费与未匹配金额口径**
- `发票基准`：手续费允许分摊展示。
- `未匹配金额` **不减手续费**（基准口径是发票总金额，不是财务记账金额）。

5) 已知易错模式（必须防）
- 发票内容是“员工餐费/餐费”，但旧线索指向“缴纳税费”的银行凭证：应优先匹配 `应付-xxxx`（如 `应付-0029`），并屏蔽与发票文本明显不一致的银行凭证。

6) **待网络填报清单 sheet（用户需求，2026-03-23 起固化）**
- 目的：把 `财务银行/财务现金/财务应付` 中“仍有剩余匹配金额”的行汇总成一个独立 sheet，供后续网络填报/报销/人工补录。
- 新增 sheet 名：`待填报`
- 固定列（严格按用户要求）：
  - `日期`
  - `发票号`：统一写 `88888888`
  - `摘要`
  - `金额`：取对应财务表 `剩余匹配金额`
  - `账目来源`：`财务银行` / `财务现金` / `财务应付`
- 摘要命名修正（参考用户提供的 2025年12月账单 `吉众带标记账单202512...xlsx`）：
  - 去掉前导 `*`
  - `购买理财` → `理财购买`
  - `宽带费_其他(194.61)` → `通信费`
  - `工资`（且来源为 `财务现金`）→ `员工-工资现金发放`
  - `_其他` 后缀移除，其余 `_` 归一为 `-`
