# 电汇专项匹配技能

## 核心规则

### 手续费（硬规则）
- 手续费 = 4.5 × N 或 5 × N（N = 1, 2, 3...）
- 候选集：{0, 4.5, 5, 9, 9.0, 10, 13.5, 15, 18, 20, 22.5, 25, 27, 30...}
- 容差：±2元（含手续费后金额相等视为匹配）

### 匹配公式
```
银行出账金额 ≈ 发票金额合计 + 手续费
```
- 发票金额合计：可以是单票，也可以是多票之和
- 手续费：4.5/5 的整数倍

### 匹配算法顺序
1. **1-sum**：一张发票 + 手续费 = 银行出账
2. **2-sum**：两张发票之和 + 手续费 = 银行出账
3. **3-sum**：三张发票之和 + 手续费 = 银行出账
4. 无解 → 标记「未匹配」

### 关键约束
- 日期不要求精确匹配（银行处理可能有延迟），以金额为准
- 手续费是必要条件：无手续费支撑的「金额接近」不算匹配
- **唯一解**：每步匹配必须唯一；有多解时标记「待人工确认」
- 已匹配发票不再参与后续匹配（贪心匹配，按银行顺序处理）

## 算法实现

```python
TOL = 2.00
FEES = sorted(set([0.0] + [round(4.5*n, 2) for n in range(1, 10)]
                              + [round(5.0*n, 2) for n in range(1, 10)]))

def fee_match(bank_amt, invoice_sum):
    """给定银行金额和发票合计，找匹配的手续费候选"""
    diff = bank_amt - invoice_sum
    if diff < -TOL:
        return None  # 银行金额不够
    for f in FEES:
        if abs(diff - f) <= TOL:
            return f
    return None

# 对每笔银行出账，枚举所有(单票/2票/3票)组合
# 找到 fee_match(bank_amt, sum_of_invoices) 有解的组合
# 若有多个解 → 待人工确认
```

## 输入数据

### ERPNext 电汇发票
- 来源：`erpnext_purchase_items_YYYYMM_v2.csv`（付款方式=电汇）
- 按发票号去重（同一发票号只取一条 grand_total）
- 关键字段：凭证号、发票号码、日期（取月-日）、金额、物料名称、备注

### 财务银行明细
- 来源：xls 财务明细的 `银行` sheet
- 取贷方（支出）行，跳过本日合计、上年结转等汇总行
- 关键字段：凭证号（如记-0001）、日期（取月-日）、摘要、贷方金额

---

## 匹配结果处理

匹配完成后，调用 `reconcile-output-format` skill 输出标准格式表格。

每张发票（每个凭证号）只写一行，但一张发票可能对应多行明细（如 LOCTITE IS 90R + 乐泰IS10E 同一发票号）。

### 匹配状态映射

| 匹配结果 | 表格标注 |
|---------|---------|
| 发票已匹配某银行凭证 | `✅ 已匹配 {财务凭证号}` |
| 发票未匹配任何银行凭证 | `❌ 未匹配` |
| 同一发票多行明细 | 每行都标注同一个 `✅ 已匹配 {财务凭证号}` |

## 无需匹配的银行支出类型（仅限电汇场景）

以下银行摘要类型，**正常就不录入ERPNext**，标记「⏭ 无需匹配」，不算未匹配：

| 摘要关键词 | 类型 |
|-----------|------|
| 短信费、通讯费、移动/联通/电信 | 通讯费 |
| 公积金 | 社保公积金 |
| 社保费 | 社保公积金 |
| 缴纳税费、税金、增值税 | 税费 |
| 取工资、发放工资、工资 | 工资 |
| 购买理财 | 投资理财 |
| 贷款还款 | 贷款 |
| 利息、手续费（小额） | 银行手续费 |
| 备用金、借支 | 备用金 |

> 注意：此规则**仅适用于银行电汇场景**，现金和应付账款另论。
